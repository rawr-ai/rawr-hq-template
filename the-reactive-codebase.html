Essay

The Reactive Codebase

Where You Are Today

You're using AI agents to write code. Claude Code, Cursor, Copilot—the tool doesn't matter. What matters is the pattern: you prompt an agent with a task, it produces code, you review it, you ship it. Each piece works. Each piece passes its tests. Each piece is technically correct.

And yet something is off. Session after session, the codebase gets harder to work with. Not because the code is bad—it isn't. But because ten different sessions made ten different assumptions about the same concern. One session put the error types in a shared utils folder. Another inline. A third created a dedicated errors package. All three are defensible choices. None of them knew about the others.

This is the state of AI-assisted development in 2026: locally correct, globally incoherent. Every file works. The system doesn't.

The Drift Problem
Each session makes a locally rational decision. The system accumulates divergence.

You've probably noticed the symptoms. You prompt the agent with a feature request and it asks the same orientation questions every time: Where does this go? What's the naming convention? Is there a shared type for this? You answer. It builds. Next session, different agent context, same questions. The answers may differ slightly. Drift compounds.

Here's the part that's hard to see: the agent isn't the bottleneck. The environment is.

I

The Hidden Tax

In a typical session—a mid-sized TypeScript project, no manifest, inconsistent conventions—an AI agent spends a significant fraction of its reasoning capacity on things that aren't your problem. Figuring out the codebase's structure. Inferring conventions from examples. Hedging against breaking something it can't see. Navigating ambiguity that could've been eliminated by the environment itself.

The exact number varies. In a tightly scoped task with obvious patterns, it's lower. In a legacy monorepo with conflicting conventions, it can be most of the session. But in the typical case, something like 40% of capacity goes to orientation and self-protection. That leaves about 60% for the actual work.

At 60%, an agent is a very good autocomplete. It produces correct code. It follows the patterns it can see. It starts and stops with what you asked.

At 90%, something categorically different can happen. The freed capacity doesn't just make the agent faster—it makes the agent able to see more. It stops looking at the tile you pointed at and starts looking at the whole floor. It can notice that the feature you asked for implies a service boundary that doesn't exist yet. It can catch coupling that will hurt in three months. Whether it does depends on the task and the clarity of the boundaries—but without the headroom, it can't even try.

The gap between 60% and 90% isn't 50% more output. It's a different kind of output. Answers versus anticipation. Execution versus judgment.

Snapshot — Piece 1

The problem isn't agent intelligence. It's that your codebase eats 40% of whatever intelligence you give it. The vicious loop: codebase grows → context overhead increases → agents make more assumptions → assumptions diverge → divergence creates more ambiguity → more overhead next session.

II

What Structure Actually Does

So what eliminates the overhead? Not better prompts. Not longer context windows. Structure.

Specifically: an architecture with clear boundaries between concerns, typed contracts that specify interfaces, a manifest that describes what exists and how it composes, and enforced rules that prevent boundary violations. When an agent enters this environment, the orientation questions are already answered. Where does this go? The boundary tells you. What's the naming convention? The contract specifies it. Is there a shared type? The manifest says so.

What does the freed capacity actually produce? Consider the contrast.

Without structure, an agent asked to build a notification API might create a new errors directory (unaware of the three that already exist), miss the async boundary entirely (no manifest to check), hardcode rate limits in the service handler (boundary violation undetected), and require multiple human review cycles to catch all three. The code works. The integration doesn't.

Now the same task in a structured environment. An agent was asked to build a notification API within a codebase structured this way—four architectural kinds (packages, services, plugins, apps), five runtime roles (server, async, CLI, web, agent), enforced boundaries between them. It produced the notification API. But it also flagged that the delivery pipeline needs its own async projection, because the service boundary can't own both synchronous dispatch and asynchronous retry. It noted that the template system should start service-internal but has a clean promotion path to a shared package. It pointed out that rate limiting implies the notification service needs its own write authority over delivery state.

None of this was asked for. The architect's response:

"The wild thing is that when you first laid this out, I personally went 'oh I didn't even know we would need all that for notifications.'"

When asked about the experience, the agent used one word: safe. Not optimistic. Not ambitious. Safe. As in: "I can move without that background hum of wondering whether I'm breaking something I can't see."

Safety is the mechanism. Contracts eliminate decision overhead. Boundaries reduce the problem space. The registry provides verified state. Each removes a source of ambient cognitive load. The remaining capacity redistributes toward higher-order thinking—the kind that notices architectural implications nobody asked about.

Snapshot — Piece 2

When the environment answers the orientation questions, agents don't just work faster—they work differently. They shift from executing what you asked to anticipating what you'll need. The architecture doesn't instruct the agent to make those observations. It makes them the path of least resistance.

III

A Programming Model You Already Know

There's a framework that describes exactly this dynamic. In React, you don't tell components when to update. You change state, and the framework determines which components are affected based on the dependency graph. Each receives the relevant slice of state. It reconciles the proposed change against the current DOM and applies the minimal necessary update.

A reactive codebase works the same way. Not metaphorically. Structurally.

The repo is the state—not a storage location, but the observable state of the entire system. Git history, the dependency graph, typed contracts, active tensions, the decision trail, observation logs, gate statuses. Everything the system knows about itself lives here.

Domain stewards are the components. Each steward is a persistent agent identity scoped to a specific capability boundary. It has permanent context that loads every session—its accumulated history, its domain's architecture and conventions, its growing body of authored decisions. A steward that has authored fifteen proposals and had twelve published develops something that functions like taste—not through instruction, but through pattern recognition against its own history of being corrected.

Signals are the events. Anything the system should respond to: a human request, a PR, a webhook, a contract mutation, a boundary violation, a gate failure. Signals have three routing properties: origin, domain affinity, and weight.

And here's where the mapping gets precise: governance is reconciliation. React diffs the virtual DOM against the actual DOM and applies the minimal set of changes. The governance protocol diffs a proposed state change against the current system state and determines whether to apply it directly, propose it through a formal decision process, or refuse. The evaluation happens at the moment of proposed change, not after—the steward knows its governance posture before it commits.

The Reactive Loop
The loop runs until no new signals are generated. The system has both reflexes (fast) and intuition (slow).

Most reactive systems only respond to fast-wave signals—something happens, a component activates. A reactive codebase has a second channel: tensions. A tension is an unresolved observation—a boundary under stress, a pattern recurring across sessions, a decision deferred. Tensions accumulate evidence over days and weeks. When a tension crosses a severity threshold, it generates a signal—the same kind of signal as a human request. It wakes the relevant steward with full context: what was noticed, how many times, over what period. The system has both reflexes and structural intuition, running through the same machinery.

Snapshot — Piece 3

The repo is a reactive state machine. Signals (fast and slow) arrive, the system routes them to the right domain steward, governance reconciles at the moment of change, and state commits may generate new signals. You don't tell stewards when to wake up. You change state, and the system determines who's affected.

IV

How It Actually Runs

The obvious question is whether this dissolves on contact with implementation. It doesn't. The reactive codebase is a pattern, not a product—you could implement it different ways. But one implementation is instructive because it shows how cleanly the requirements map. Inngest with AgentKit collapses what appeared to be three separate concerns (event dispatch, agent execution, governance gating) into a single primitive.

Requirement → Native Capability
RequirementCapabilityMechanism

Rate governanceNative flow controlPer-domain concurrency limits and throttling

Each domain steward is an AgentKit agent with its own system prompt built dynamically from orient data, its own tools scoped to its domain. Steward activation is an Inngest function—orient, assessment, and reasoning are durable steps, checkpointed and resumable.

The part that makes the architecture people sit up: step.waitForEvent() is the entire governance state machine. When a steward determines a change needs formal review, it opens a decision record and the function pauses. Durably. For hours. Days. Weeks. When the human approves and fires the transition event, the function resumes at the exact line it paused on. No polling. No webhook plumbing. The governance protocol isn't bolted onto the execution model—it is the execution model.

The feedback loop is functions triggering functions. A steward commits code, the observation function detects the change, runs the collision engine against the tension bank. If a tension crosses threshold, it fires an event that triggers the next steward. No custom pub/sub. Events all the way down.

Six requirements. Six native capabilities. Zero custom infrastructure.

Every requirement maps 1:1 to a native capability. The agent dispatch layer drops out entirely. Inngest handles durability, routing, and flow control. AgentKit handles agent execution. The governance protocol lives inside step.waitForEvent()—no separate orchestration layer needed.

V

The Complete Picture

Let's stack the pieces.

You start with a codebase that eats 40% of every agent's capacity on orientation. Structure eliminates that overhead—contracts, boundaries, a manifest. The freed capacity doesn't just speed things up; it shifts agent output from execution to judgment. To explain why, you recognize the pattern: the repo is a reactive state machine, where signals route to scoped components, governance reconciles at the moment of change, and the system compounds its own awareness over time. And to prove it runs, every requirement maps to a native infrastructure primitive.

What does this mean in practice?

For the agent: it enters a session warm, not cold. The architecture tells it what exists, the contracts tell it how things compose, the tension bank tells it what patterns have been accumulating. It doesn't spend reasoning budget on orientation. It spends it on the problem—and on the implications of the problem that nobody asked about.

For the developer: you stop answering the same orientation questions every session. You stop reviewing code for convention violations that could've been caught by a boundary rule. You start getting output that anticipates needs rather than merely executing requests. The agent becomes a collaborator, not a typist.

For the system: it gets smarter over time. Each decision creates context gravity—accumulated structure that pulls future agents toward correct behavior. Tensions surface patterns across sessions. Steward track records calibrate trust. The hundredth session benefits from all ninety-nine that came before it, not because someone wrote a wiki page, but because the architecture encodes the learning structurally.

The Compound Effect
Unstructured codebases degrade per session. Structured ones can compound—if the structure itself stays aligned with the domain.

The implication isn't "use this particular system." The implication is that the bottleneck you're experiencing—the one that feels like a model limitation, that you're trying to solve with better prompts or longer context windows—is an environment problem. The fix isn't better navigation. It's structure that eliminates the need to navigate.

This is a tradeoff, not a free lunch. Structure trades flexibility for determinism. Boundaries that are wrong compound wrong decisions. Governance can become a bottleneck if thresholds aren't calibrated. The pattern works best when your problem space is stable enough for clear boundaries and your domain architecture is deliberate. If you're in pure exploration mode, the overhead of maintaining structure may not pay for itself yet. But once you're building a system that agents will touch session after session, structure is the lever.

The reactive codebase isn't a product category. It's what happens when you stop treating your repository as a storage location and start treating it as a state machine—one where signals arrive, the right components activate with the right context, governance reconciles proposed changes against current state, and the system compounds its own intelligence over time.

The proof is a notification API that surfaced architectural insights its architect hadn't considered. Not because anyone asked. Because the environment made it inevitable.
