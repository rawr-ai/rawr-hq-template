## Investigator 1: durable identity, ingress, retries

Research query: For RAWR HQ runtime realization, which Inngest primitives and operational semantics should the plugin/runtime specs encode for durable plugin workflows?

### Bottom-line position

RAWR should encode Inngest durable plugin workflows as a runtime-owned function/step contract, not as arbitrary plugin background code. The minimum normative surface is: stable `createFunction` identity, explicit trigger event names, stable step IDs, all nondeterministic side effects inside `step.run`, runtime-owned `/api/inngest` or Connect worker ingress, environment-scoped signing/event keys, explicit retry/idempotency/rollback policy, and layered observation that keeps HTTP/protocol success, Inngest run/step state, and RAWR payload status separate. The corpus supports these as spec requirements with high confidence; it does not yet support claiming RAWR has proven durable scheduling, retry replay, or production ingress behavior without a Dev Server, self-hosted, or production-like Inngest control-plane gate.

### Evidence and source URLs

- Durable function identity starts at `createFunction`. Inngest requires a function `id` as "a unique identifier for your function" that "should not change between deploys"; the same configuration object carries `triggers`, `concurrency`, `throttle`, `idempotency`, `rateLimit`, `retries`, `onFailure`, `cancelOn`, and `timeouts` fields. Source: https://www.inngest.com/docs/reference/typescript/functions/create.
- Step identity is the durability key. The steps docs say the first argument of every step method is an `id`, each step can be retried/debugged/recovered independently, and Inngest uses the ID to memoize step state across function versions. The same page says each step executes as a separate HTTP request, nondeterministic logic such as DB/API calls belongs inside `step.run`, and a successful `step.run` result is saved so the step will not re-run. Source: https://www.inngest.com/docs/learn/inngest-steps.
- Durable execution is step-by-step memoization with external state, not a local TypeScript control-flow guarantee. Inngest persists step results in a managed function state store, memoizes successful steps, re-executes the handler with prior state injected, skips completed steps, serializes step errors, logs attempts, and resumes from the failed step when retries remain. Source: https://www.inngest.com/docs/learn/how-functions-are-executed.
- Runtime ingress for `serve()` is one endpoint, not a plugin route. `serve()` exposes application functions via HTTP from one Inngest client and an explicit function array; Inngest recommends `/api/inngest`; the endpoint handles `GET` metadata/dev landing page, `POST` invocation, and `PUT` registration/sync; signing-key configuration belongs on the Inngest client, preferably via `INNGEST_SIGNING_KEY`, not hard-coded in `serve()`. Source: https://www.inngest.com/docs/reference/serve.
- Production ingress security is environment-keyed. Inngest signs requests between platform and app with an environment-unique signing key; SDK serve adapters verify signatures with embedded timestamps to reject replay; signing keys also authenticate API interactions and Connect workers; event/API/signing keys can be rotated; firewall allowlisting of Inngest outbound IPs is recommended. Source: https://www.inngest.com/docs/learn/security.
- Connect is a production topology option, but not yet a baseline primitive RAWR can require without caveat. It is Public Beta, uses outbound persistent WebSocket connections, supports horizontal worker scaling, automatic reconnects, graceful shutdown on `SIGTERM`, app version routing for rolling deploys, instance identity, and worker concurrency controls. It requires a long-running server and does not support serverless runtimes; local development can use `INNGEST_DEV=1` with no signing/event keys. Source: https://www.inngest.com/docs/setup/connect.
- Retry semantics distinguish errors from failures. Inngest retries errors automatically; when retries are exhausted, the step fails and will not be attempted again in that run; unhandled failure bubbles to function failure and cancels future executions. Retried step bodies must be idempotent; the docs explicitly contrast unsafe insert with safe upsert. Source: https://www.inngest.com/docs/guides/error-handling.
- Rollback/fallback behavior is a step-level programming contract. A failed step that exhausts retries throws `StepError`, which can be caught to recover, fall back, ignore a non-critical step, or run a compensating `step.run` rollback; unhandled failures bubble to the function and mark it failed. Source: https://www.inngest.com/docs/features/inngest-functions/error-retries/rollbacks.
- Durable proof requires an Inngest control plane/state store, not only SDK constructibility. Self-hosting architecture includes Event API, Event stream, Runner, Queue, Executor, State store, Database, API, and Dashboard UI; Runner schedules runs, resumes `waitForEvent`, cancels `cancelOn`, and writes event history; Executor executes functions/steps, writes incremental run state, and retries failures. Source: https://www.inngest.com/docs/self-hosting.

### Concrete spec implications

- Plugin workflow declarations should include stable fields: `workflow.id` mapping to Inngest function `id`, `workflow.name` as display-only, `triggers[]` with event names and optional CEL `if` predicates, `steps[]` with stable `id`, declared side-effect category, idempotency key/dedup strategy, retry policy, timeout policy, and rollback/fallback behavior.
- RAWR should reject or flag workflow definitions where function IDs or step IDs are generated from runtime data, plugin install path, timestamps, deploy hashes, array indices that can reorder, or human-readable labels likely to be edited. Renames should require an explicit migration/versioning annotation because Inngest uses step IDs for memoized state across function versions.
- The plugin spec should treat top-level function-handler code as deterministic orchestration only. Allowed top-level work: read event payload, branch on stable data, compose `step.*` calls, and return assembled results. Disallowed top-level work: DB writes, external API calls, event sending via non-step APIs, filesystem mutation, network calls, random/time-dependent decisions that affect side effects, and direct plugin state mutation.
- All side effects should cross a `step.run` or other durable step primitive. For event emission from inside a function, RAWR should prefer `step.sendEvent` over raw `inngest.send()` because Inngest documents it as reliable event delivery from within functions. Source: https://www.inngest.com/docs/reference/typescript/functions/create.
- Runtime specs should own the Inngest client, function registry, `/api/inngest` serve endpoint, signing key lookup, event key lookup, app ID, serve origin/path, sync behavior, app version, instance ID, Connect/serve topology, and observability wiring. Plugin declarations should contribute workflow definitions and policy metadata, not expose their own browser-facing Inngest routes.
- `/api/inngest` should be documented as runtime/control-plane ingress only. Browser and ordinary plugin API callers must not be guided to call it. Allowed callers are Inngest Cloud/self-host control plane, local Dev Server/self-host in local mode, and runtime sync/admin paths with signing semantics.
- Specs should distinguish `serve()` and Connect as topology choices. `serve()` is the default portable HTTP endpoint for serverless or ordinary app hosting. Connect is better for long-running container workers, lower latency, horizontal scaling, no inbound load balancer, and steps not bound by platform HTTP timeout, but because the captured docs label it Public Beta, RAWR should model it as an operator/runtime deployment option rather than a required plugin primitive.
- Production policy should require `INNGEST_SIGNING_KEY` and `INNGEST_EVENT_KEY` per environment, key rotation procedure, no hard-coded secrets, optional `INNGEST_BASE_URL` for self-hosting, and firewall/IP allowlisting where the deployment stack supports it. Local-dev exceptions such as `INNGEST_DEV=1` and no keys must be called out as non-production behavior.
- Retry policy should be explicit. Function declarations should default consciously to Inngest's documented retry default only if RAWR accepts it; otherwise declare `retries` in the plugin/runtime spec. The valid Inngest range in the captured create docs is `0` to `20`, with default `4`.
- Idempotency is a step-body obligation as much as a function config option. RAWR should require an idempotency note for every side-effecting step: e.g. use upsert/deduplication keys for writes, natural external idempotency keys for third-party calls, and audit-event IDs for event publication. Unsafe "insert every retry" patterns should fail spec review unless wrapped by an external uniqueness guarantee.
- Failure taxonomy should be layered: transport/HTTP status from `serve()` or Connect, Inngest function run status, Inngest step status, retry attempt number, exhausted retry/`StepError`, handled rollback/fallback result, unhandled function failure, and RAWR payload-level success/failure. RAWR should not compress these into one boolean or one HTTP status.
- Observation/tests should be layered as well: pure conformance tests can verify stable IDs, no-context-smuggling, function registry shape, route ownership, and policy projection; only a Dev Server/self-host/Cloud-backed gate can verify memoized resume, retry attempts, StepError behavior, `onFailure`, cancellation, wait resumption, run history, and production key/sync behavior.

### Unresolved caveats

- The current captured corpus is official Inngest documentation plus width claims; it is enough to prescribe the spec surface, but not enough to prove RAWR's runtime has executed a durable retry/resume path. That needs a local Dev Server, self-hosted Inngest, or Cloud-backed integration gate. Sources: https://www.inngest.com/docs/learn/how-functions-are-executed and https://www.inngest.com/docs/self-hosting.
- Connect should not be over-promoted yet. It has attractive topology semantics for RAWR workers, but the captured docs mark it Public Beta and impose long-running-server/runtime requirements. RAWR should keep `serve()` as the conservative baseline while allowing Connect as an operator-selected topology. Source: https://www.inngest.com/docs/setup/connect.
- The docs give the `retries` numeric range and default, but they do not by themselves decide RAWR's domain-specific defaults per plugin class. RAWR still needs policy decisions for must-process workflows, user-visible workflows, fire-and-forget telemetry, and compensating actions. Source: https://www.inngest.com/docs/reference/typescript/functions/create.
- Step ID migration needs care. Inngest docs establish why stable step IDs matter, but this packet did not fetch a dedicated Inngest versioning guide. RAWR should treat function/step renames as breaking durable-state changes until a migration/versioning policy is explicitly designed. Sources: https://www.inngest.com/docs/learn/inngest-steps and https://www.inngest.com/docs/learn/how-functions-are-executed.
- Security posture has environment-key, signing, timestamp replay rejection, and rotation guidance, but firewall allowlisting depends on deployment platform support and current Inngest IP ranges. The packet did not capture the live IP list pages, so the spec should reference them rather than hard-code ranges. Source: https://www.inngest.com/docs/learn/security.
- Self-host defaults are convenient but not production-scale proof. The captured docs say default self-hosting uses SQLite plus an in-memory Redis server for queue/state store with snapshots, while external Redis/Postgres are the reliability/scaling path. RAWR should not call a default single-node self-host proof equivalent to a production topology. Source: https://www.inngest.com/docs/self-hosting.

### Committed position

The evidence adds up to a firm spec stance: RAWR's durable plugin workflow contract should be Inngest-shaped at the function, trigger, step, ingress, retry, and observation layers, while production topology and operator controls stay runtime-owned. The load-bearing reason is that Inngest durability is keyed by stable IDs and persisted step state outside the handler; if RAWR lets plugins hide identity, ingress, side effects, or status semantics behind arbitrary async code, it destroys the very operational guarantees it is trying to encode.
