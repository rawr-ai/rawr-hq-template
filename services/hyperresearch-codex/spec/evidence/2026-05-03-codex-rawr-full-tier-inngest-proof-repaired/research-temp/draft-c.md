## I. Executive Summary

RAWR HQ should encode durable plugin workflows as a narrow Inngest-shaped contract: plugins declare stable workflow identity, trigger events, step IDs, retry/idempotency/failure policy, waits, batching intent, and flow-control intent; the runtime owns the Inngest client, function registry, `/api/inngest` ingress, signing/event keys, app sync, topology, observability, and production proof. The current RAWR runtime-realization evidence supports a contained boundary claim around `serve()`, `createFunction`, and `step.run`, but it does not prove durable scheduling, memoized resume, retry replay, wait resumption, run history, or production ingress. Those claims require a live Inngest Dev Server, Cloud, or self-hosted control-plane gate. [[serve-inngest-documentation]] [[create-function-inngest-documentation]] [[inngest-steps-inngest-documentation]] [[how-inngest-functions-are-executed-durable-execution-inngest-documentation]]

The most important boundary rule is that plugin authors should never treat Inngest as arbitrary async code. `createFunction` IDs are deploy-stable compatibility keys; step IDs are durable state keys; `step.run` is the side-effect and retry boundary; `step.waitForEvent`, batching, retries, and flow control change operational semantics and therefore must be explicit spec fields rather than incidental implementation options. [[create-function-inngest-documentation]] [[inngest-steps-inngest-documentation]] [[wait-for-an-event-inngest-documentation]] [[batching-events-inngest-documentation]] If RAWR hides those choices behind plugin-local handlers, it loses the only durable guarantees worth adopting Inngest for.

The production caveat is equally firm: `/api/inngest` is runtime/control-plane ingress, not a browser-facing plugin route. Production requires environment-specific signing and event keys, SDK signature verification, app sync, endpoint reachability, secret rotation posture, and a topology choice between HTTP `serve()` and outbound `connect` workers. Local dev behavior, including dummy event keys and disabled signature verification, is useful for tests but is not production evidence. [[security-inngest-documentation]] [[local-development-inngest-documentation]] [[connect-inngest-documentation]] [[render-inngest-documentation]]

## II. Plugin Spec Surface: What Authors May Declare

Plugin workflow specs should expose only the parts of Inngest semantics that are part of the durable plugin contract. A workflow declaration should include a stable `workflow.id` that maps to the Inngest function `id`, a display name that is explicitly non-durable, trigger event names or cron schedules, optional trigger predicates, and a declared handler shape. The Inngest docs make the function `id` required and say it should not change between deploys; that turns a plugin workflow ID into a compatibility key, not a label. [[create-function-inngest-documentation]]

Each workflow should then declare a set of durable steps with stable `step.id` values, side-effect categories, input/output contracts, idempotency strategy, retry policy, timeout policy, and rollback or fallback behavior. Inngest uses the first argument of step methods as the step identity and memoizes step state around that ID; successful `step.run` output is saved and not re-run. That means a step rename is not a harmless refactor. RAWR should treat function ID and step ID renames as breaking durable-state changes unless a versioning or migration annotation is present. [[inngest-steps-inngest-documentation]] [[how-inngest-functions-are-executed-durable-execution-inngest-documentation]]

The plugin authoring grammar should permit these workflow primitives:

- `trigger.event` and `trigger.cron`, with optional CEL (Common Expression Language) predicates where the runtime can validate and project them.
- `step.run` for side-effecting work and retriable deterministic units of work.
- `step.waitForEvent` for event-resumed continuations with required event name, timeout, match predicate, and timeout branch.
- `step.sendEvent` for reliable event publication from inside a function, rather than raw event sending in handler code.
- `batchEvents` only for batch-safe workflows with explicit `maxSize`, `timeout`, optional `key`, optional `if`, and handler input shape.
- Function-level `retries`, `onFailure`, `cancelOn`, `timeouts`, and flow-control policy where RAWR can enforce the semantics. [[create-function-inngest-documentation]] [[inngest-steps-inngest-documentation]] [[wait-for-an-event-inngest-documentation]] [[batching-events-inngest-documentation]]

That surface should deliberately exclude runtime topology fields. Plugin specs should not own the Inngest client, `serve()` adapter, signing keys, event keys, endpoint path, endpoint origin, app version, worker instance ID, Connect gateway URL, or sync mechanism. Those are deployment and runtime responsibilities. The plugin contributes durable intent; the runtime turns that intent into an Inngest function registry. [[serve-inngest-documentation]] [[security-inngest-documentation]] [[connect-inngest-documentation]]

## III. Runtime-Owned Boundary: Ingress, Registry, Keys, and Topology

The runtime should own a single Inngest app boundary: one client, one registry of plugin-contributed functions, and one configured ingress path. `serve()` takes an Inngest client and an explicit array of functions, exposes the functions over HTTP, and is documented around a recommended `/api/inngest` path. The endpoint has control-plane method semantics: `GET` returns metadata and a development landing page, `POST` invokes functions with run state, and `PUT` registers or syncs functions. That is not a normal application API and not a plugin extension route. [[serve-inngest-documentation]]

RAWR specs should therefore state that `/api/inngest` is runtime-owned ingress. Allowed callers are Inngest Cloud, a self-hosted Inngest server, the local Dev Server, or runtime sync tooling operating under signing semantics. Browser clients, ordinary plugin APIs, and user-originated HTTP traffic should not be pointed at this endpoint. This matters because the serve adapter is the signature-verification and control-plane handoff point; collapsing it into the plugin API surface would blur security, sync, and invocation responsibilities. [[serve-inngest-documentation]] [[security-inngest-documentation]]

Key ownership is also runtime-owned. Production requires `INNGEST_SIGNING_KEY` for signed platform-to-app requests and `INNGEST_EVENT_KEY` for sending events. Inngest signs requests with an environment-specific signing key, SDK adapters verify signatures and timestamps to reject replay, and keys should be kept secret and rotated through operational tools. Plugin authors should never hard-code keys, request signing keys, or choose environment key material. They may declare that a workflow emits or consumes events, but the runtime decides how event delivery is authenticated in each environment. [[security-inngest-documentation]] [[render-inngest-documentation]]

Topology belongs to the runtime/operator layer. `serve()` is the conservative baseline for HTTP-hosted functions, especially where the app already has an inbound endpoint. `connect` is a different topology: outbound persistent worker connections over WebSockets, horizontal scaling, graceful shutdown, app versions, instance IDs, worker concurrency, health checks, and long-running-server requirements. The captured docs mark Connect as Public Beta and say serverless runtimes are not supported, so RAWR should expose it as an operator-selected deployment mode, not a plugin primitive or default production guarantee. [[connect-inngest-documentation]] [[serve-inngest-documentation]]

For self-hosted or local control-plane deployments, the runtime still owns the durable substrate. Inngest self-hosting includes Event API, event stream, Runner, Queue, Executor, state store, database, API, and UI; these are the pieces that schedule runs, persist step output and errors, resume waits, apply flow control, and retry failures. A local package passing through `createFunction` and `step.run` without that substrate proves shape and boundary compatibility, not durable execution. [[self-hosting-inngest-documentation]] [[how-inngest-functions-are-executed-durable-execution-inngest-documentation]]

## IV. Step Semantics: Where Plugin Code May Actually Do Work

RAWR should make top-level Inngest handlers deterministic orchestration zones. At handler scope, plugin code may inspect the triggering event, branch on stable event data, compose `step.*` calls, assemble return values, and log through the runtime logger. It should not perform database writes, network calls, filesystem mutation, event publication through non-step APIs, random/time-dependent decisions that affect side effects, or direct mutation of plugin state. The reason is not stylistic. Inngest executes functions incrementally and re-enters handlers while injecting memoized state for completed steps. [[how-inngest-functions-are-executed-durable-execution-inngest-documentation]]

Every nondeterministic side effect should cross a durable step boundary. The steps docs say each step executes as a separate HTTP request and explicitly place DB/API calls inside `step.run`; the durable execution docs explain that successful step results are persisted, memoized, and skipped on later executions. So the plugin authoring rule should be simple: if the code changes the outside world, calls another system, depends on time/randomness, or emits a durable event, it must be inside `step.run`, `step.sendEvent`, `step.invoke`, or another approved step primitive. [[inngest-steps-inngest-documentation]] [[how-inngest-functions-are-executed-durable-execution-inngest-documentation]]

This implies a concrete review rule for plugin workflow specs. A side-effecting step must include:

- A stable step ID that is not generated from runtime data, install path, array index, deploy hash, timestamp, or editable label.
- An idempotency explanation, such as upsert keys, external idempotency keys, event IDs, or uniqueness constraints.
- A retry count or accepted runtime default.
- A timeout or reason no timeout is needed.
- Failure handling: propagate, catch `StepError`, fallback, rollback, mark non-critical, or trigger `onFailure`.
- Observability labels that let RAWR correlate plugin workflow ID, Inngest function ID, step ID, run ID, attempt, and RAWR payload state. [[errors-retries-inngest-documentation]] [[rollbacks-inngest-documentation]] [[create-function-inngest-documentation]]

The most common authoring mistake will be treating `step.run` as just a wrapper. It is the durable transaction boundary. Inngest retries thrown errors, persists successful return values, and reuses those values on resume. Therefore a step body must be safe to run more than once until it succeeds, and its success value must be safe to store in Inngest state. If a step output can contain regulated or sensitive data, the runtime spec needs an encryption or redaction policy because Inngest stores step output and function output as function state. [[errors-retries-inngest-documentation]] [[security-inngest-documentation]]

## V. Retry, Error, and Failure Semantics

RAWR should encode retries as a workflow contract, not an implicit background behavior. Inngest distinguishes an error from a failure: an error causes retry; exhausting retries makes the step fail; an unhandled failed step bubbles to function failure and cancels future executions. The `createFunction` reference exposes `retries` from `0` to `20`, with a documented default of `4`, and also exposes `onFailure`. RAWR can accept the Inngest default, but it should do so explicitly per workflow class rather than by accident. [[errors-retries-inngest-documentation]] [[create-function-inngest-documentation]]

Idempotency should be mandatory for every retriable side-effecting step. The official error docs give the core reason: re-running a step on error requires code that can safely run multiple times; inserting a new user repeatedly is unsafe, while upserting a user is the intended pattern. RAWR should turn that into a plugin review gate. A plugin workflow that writes to storage, calls an external API, or emits an event without a deduplication or idempotency mechanism should fail conformance unless the workflow explicitly declares that duplicate side effects are acceptable. [[errors-retries-inngest-documentation]]

Failure handling should be layered. A failed `step.run` after exhausted retries can throw `StepError`, which may be caught to recover, ignore a non-critical step, fall back to another provider, or run a compensating rollback step. RAWR specs should distinguish these outcomes from unhandled function failure. A plugin workflow that uses fallback should declare whether the fallback is semantically equivalent, degraded-but-successful, or a separate RAWR payload failure wrapped inside a successful Inngest run. [[rollbacks-inngest-documentation]]

That distinction matters because RAWR already has evidence of layer disagreement: a transport or protocol success can contain a runtime failure payload. The durable workflow spec should preserve at least five layers of status: HTTP/control-plane result, Inngest run status, step status, retry/attempt status, and RAWR payload status. A single boolean like `success` is too lossy. The runtime should record `runId` and `attempt`, expose step IDs in logs, and map plugin-level results separately from Inngest execution results. [[create-function-inngest-documentation]] [[how-inngest-functions-are-executed-durable-execution-inngest-documentation]]

## VI. Continuations, Batching, and Flow Control

`step.waitForEvent` should be exposed as an explicit continuation contract. A wait declaration needs an awaited event name, CEL match predicate, timeout, timeout branch behavior, and race mitigation. Inngest resumes a wait when a matching event arrives and returns `null` on timeout, but the docs also warn that listening starts when the wait is executed; events sent before that point are not handled. For human-in-the-loop and cross-plugin workflows, RAWR should require a pre-wait state check or event-ordering proof so a fast external event cannot be lost before the wait is reached. [[wait-for-an-event-inngest-documentation]]

The plugin-facing contract for waits should also state why the workflow is waiting. Event waits are appropriate for approvals, external system completion, cross-plugin joins, and audited user actions because events fan out, decouple code, and preserve an event trail. They are not a replacement for simple time delays; `step.sleep` exists for that. A plugin spec that waits for another event should identify whether timeout means failure, degraded continuation, retryable absence, cancellation, or a domain-specific no-op. [[wait-for-an-event-inngest-documentation]] [[inngest-steps-inngest-documentation]]

Batching should be opt-in and validated. `batchEvents` changes handler shape from a single `event` to an `events` array, waits for `maxSize` or `timeout`, can group by `key`, can conditionally batch via `if`, and has a 10 MiB safety limit. It is useful for batch database writes or third-party APIs that accept bulk operations, but it changes correctness semantics. RAWR should require plugin authors to prove that the operation is batch-safe and that partial batch behavior, memory use, and ordering expectations are acceptable. [[batching-events-inngest-documentation]] [[create-function-inngest-documentation]]

Batching also needs hard incompatibility validation. The captured docs state batching cannot be combined with idempotency, rate limiting, cancellation events, or priority, and custom concurrency keys are evaluated against the first event in a batch unless batch and concurrency keys align. RAWR should make these invalid combinations fail spec validation rather than letting them surface as runtime surprises. If a workflow is must-process and per-event idempotency matters, batching may be the wrong primitive unless RAWR adds its own per-event deduplication inside the batch step. [[batching-events-inngest-documentation]]

Flow-control terms should not be used interchangeably. Concurrency limits executing steps across runs; throttling limits the rate at which new runs start and queues excess runs FIFO; rate limiting imposes a hard limit and skips excess matching events; debounce coalesces bursty events over a window; priority changes execution ordering. The plugin spec should expose these as distinct policies with lossy/non-lossy labels. Rate limiting is unsafe for must-process workflows unless skipping is explicitly desired; throttling is the better fit when all work must eventually run but starts need smoothing. [[flow-control-inngest-documentation]] [[throttling-inngest-documentation]] [[rate-limiting-inngest-documentation]]

The runtime should reserve override authority for platform protection. A plugin may request `concurrency`, `throttle`, `rateLimit`, `debounce`, `priority`, `batchEvents`, and timeouts, but the runtime/operator layer should cap or reject policies that can overload shared resources, silently drop required events, conflict with batching, or exceed account/deployment limits. This is where plugin authoring and operations meet: authors declare intent; runtime validates it against platform guarantees. [[create-function-inngest-documentation]] [[flow-control-inngest-documentation]]

## VII. Testing and Proof Gates

RAWR should split testing into three gates. The first is pure spec/conformance testing: verify stable function IDs, stable step IDs, allowed primitive usage, deterministic top-level handlers, route ownership, policy projection, no hard-coded secrets, and invalid flow-control combinations. This gate can be hermetic and does not prove durability; it proves that plugin declarations can be projected into an Inngest-shaped registry. [[create-function-inngest-documentation]] [[serve-inngest-documentation]]

The second gate is a vendor-local or self-hosted durable execution test. The Inngest Dev Server can be pointed at a local `/api/inngest`, auto-discover common endpoints, invoke functions, send test events, and show function behavior locally. Self-hosting gives an explicit Event API, Runner, Queue, Executor, state store, and UI. This gate should prove at least one memoized step skip on resume, one failing step that retries and then succeeds or exhausts into handled failure, one `waitForEvent` matched path, one timeout path, one batched workflow if batching is supported, and visible run/step history. [[local-development-inngest-documentation]] [[self-hosting-inngest-documentation]] [[how-inngest-functions-are-executed-durable-execution-inngest-documentation]]

The third gate is pre-production or production-like ingress/security testing. It should verify real signing keys, event keys, app sync, `/api/inngest` reachability, signature verification, key separation by environment, `serveOrigin` or `servePath` behavior where proxies are involved, deployment sync after function changes, and observation of run IDs and attempts. Render is only one captured deployment example, but its guidance is enough to show the pattern: deploy the app that serves functions, set signing and event keys, and sync the app with Inngest. [[security-inngest-documentation]] [[render-inngest-documentation]] [[serve-inngest-documentation]]

Local development must stay in its lane. The Dev Server is useful and can run in testing or CI, but local mode allows dummy event keys and `INNGEST_DEV=1` disables signature verification in the TypeScript SDK path described by the docs. A passing local test proves workflow shape and local durable behavior against a development control plane; it does not prove production ingress security, production key handling, app sync under real environments, firewall allowlisting, or hosted platform behavior. [[local-development-inngest-documentation]] [[security-inngest-documentation]]

Production claims should require captured artifacts, not prose assertions. Before RAWR says a durable plugin workflow is production-ready, the evidence should include the generated function registry, endpoint debug output, environment key configuration, app sync record, event payloads, run IDs, attempt counts, step histories, retry/failure histories, wait histories, and logs that correlate plugin workflow IDs to Inngest run IDs. Without those artifacts, the honest claim is "spec-compatible and locally testable," not "durably proven in production." [[how-inngest-functions-are-executed-durable-execution-inngest-documentation]] [[self-hosting-inngest-documentation]]

## VIII. Production Caveats and Future Implications

The immediate production caveat is that RAWR should not promote the current contained runtime-realization proof into a durable workflow proof. The available evidence supports spec design: how plugin declarations should be shaped, what runtime should own, and which gates should be required. It does not support a claim that RAWR has already demonstrated persistent state replay, retry exhaustion, wait resumption, production signing, or topology correctness. That boundary should be written directly into the plugin/runtime specs so future agents do not turn constructibility into operational certainty. [[how-inngest-functions-are-executed-durable-execution-inngest-documentation]] [[self-hosting-inngest-documentation]]

The second caveat is migration. Inngest docs captured here strongly support stable function and step IDs, but they do not provide a full versioning migration policy. RAWR should therefore treat function ID, trigger, step ID, step order, batching shape, and flow-control changes as compatibility-sensitive until a dedicated migration rule exists. A reasonable interim policy is dual registration for changed workflow IDs, explicit version suffixes for breaking workflow changes, and a refusal to silently rename step IDs in workflows that may have in-flight runs. [[create-function-inngest-documentation]] [[inngest-steps-inngest-documentation]]

The forward-looking implication is that plugin authoring should become more declarative, not more permissive. As RAWR adds durable workflows, the plugin API should move toward a manifest-like workflow declaration that can be statically inspected, projected into Inngest functions, checked for invalid policy combinations, and tested against a reference runtime. This will make plugin workflows less "write any async code" and more "declare a durable workflow contract the runtime can enforce." That tradeoff is correct: durable execution only helps if the runtime can see and govern identity, side effects, retries, waits, batching, and flow control. [[create-function-inngest-documentation]] [[flow-control-inngest-documentation]]

Over time, RAWR can support multiple execution topologies behind the same plugin contract. `serve()` can remain the baseline HTTP ingress, while Connect can become a runtime/operator option for containerized workers that need outbound connections, horizontal scaling, graceful shutdown, app versions, instance IDs, and worker concurrency. Self-hosted Inngest can become a stronger local or private-control-plane gate when RAWR needs to inspect queue/state behavior directly. But plugin specs should not change across those topology choices; the runtime adapter should. [[serve-inngest-documentation]] [[connect-inngest-documentation]] [[self-hosting-inngest-documentation]]

The committed answer is that RAWR durable plugin workflows should encode Inngest semantics at the contract edges and keep operations centralized. Expose stable function IDs, trigger events, stable step IDs, `step.run` side-effect boundaries, retry/idempotency/failure policy, waits, batching, and flow-control intent to plugin authors. Keep `/api/inngest`, keys, sync, topology, and production evidence in the runtime. Require durable control-plane proof before production claims. That is the only boundary that preserves both plugin author ergonomics and the operational semantics Inngest is meant to provide.
