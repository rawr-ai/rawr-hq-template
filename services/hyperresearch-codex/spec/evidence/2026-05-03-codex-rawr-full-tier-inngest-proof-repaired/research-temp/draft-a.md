# Draft A: Service/Spec Architecture Reading

## I. Executive Summary

RAWR HQ should encode Inngest durable plugin workflows as a runtime-owned service contract with plugin-declared workflow metadata. The minimum spec surface is: one runtime-owned Inngest client and function registry, a controlled `serve()` ingress at `/api/inngest` or an explicit Connect worker topology, stable `createFunction` IDs and trigger names, stable step IDs, all nondeterministic side effects inside `step.run` or another durable step primitive, explicit retry/idempotency/failure policy, event-wait contracts, batching and flow-control validation, and separate local, integration, and production proof gates. This is the architectural line: plugins declare durable workflow intent; the runtime owns ingress, keys, function registration, topology, and observation.

The RAWR runtime-realization evidence should be read as boundary and conformance evidence, not as production durability proof. It supports a spec direction, but it does not prove memoized resume, retry replay, `waitForEvent` resumption, production signing, app sync, or run history without a real Inngest Dev Server, Cloud, or self-hosted control plane. Inngest durable execution depends on persisted state outside the handler, independently retried steps, and handler re-entry with prior step results injected, so RAWR must not treat TypeScript constructibility or a mounted endpoint as sufficient proof of durable plugin workflows [[how-inngest-functions-are-executed-durable-execution-inngest-documentation]] [[self-hosting-inngest-documentation]].

The practical answer is therefore a two-layer spec. Plugin specs should contribute stable workflow IDs, trigger events, step IDs, side-effect declarations, retry/idempotency requirements, wait predicates, batch/control policy requests, and failure behavior. Runtime specs should own the Inngest client, `/api/inngest` handler, function array, signing and event keys, environment mapping, app sync, Connect versus `serve()` deployment choice, worker health, and production observation [[serve-inngest-documentation]] [[create-function-inngest-documentation]].

## II. Runtime Ingress Must Be A Service Boundary

`serve()` is not a plugin API route. It is the HTTP API handler that exposes an application's Inngest functions so Inngest can read configuration and invoke function code. The captured `serve()` docs require a client and a function array, recommend the path `/api/inngest`, and describe one endpoint that handles metadata/dev landing behavior by `GET`, function invocation by `POST`, and registration/sync by `PUT` [[serve-inngest-documentation]]. RAWR should encode that as runtime ingress: an ingress is the runtime boundary through which a control plane calls into RAWR-owned compute.

That boundary has three immediate consequences. First, plugins should not mount their own Inngest endpoints or tell browsers, ordinary plugin APIs, or workspace clients to call `/api/inngest`. Second, the runtime should build the function registry from accepted plugin workflow declarations and pass that registry to `serve({ client, functions })`. Third, route configuration such as `servePath`, `serveOrigin`, streaming, and app endpoint identity belongs to runtime deployment policy, because reverse proxies, host inference, and platform timeouts are operator concerns [[serve-inngest-documentation]].

The security model reinforces the same boundary. Inngest signs requests between platform and application with an environment-unique signing key; SDK serve adapters verify signatures and embedded timestamps to reject replay; signing keys also authenticate API interactions and Connect workers [[security-inngest-documentation]]. RAWR should require `INNGEST_SIGNING_KEY` and `INNGEST_EVENT_KEY` per environment, support rotation, avoid hard-coded secrets, and make local exceptions explicit. A local dummy event key or dev-mode signing bypass is a development convenience, not evidence that production ingress is secure [[local-development-inngest-documentation]] [[security-inngest-documentation]].

`serve()` should be the conservative default topology because it works as an HTTP endpoint on ordinary serverless or web platforms. Connect should be modeled as a runtime/operator option for long-running worker deployments, not as a plugin primitive. Connect uses outbound persistent WebSocket connections, supports horizontal workers, graceful shutdown, app version routing, instance IDs, and worker concurrency, but the captured docs label it Public Beta and exclude serverless runtimes [[connect-inngest-documentation]]. RAWR can expose a topology switch in runtime configuration; plugin workflow declarations should not depend on whether invocation arrives by `serve()` or Connect.

## III. Workflow Declarations Need Stable Durable Identity

The first durable contract is function identity. `createFunction` requires an `id` that is unique and should not change between deploys; its configuration also carries triggers and operational policy fields including concurrency, throttle, idempotency, rate limit, debounce, priority, batching, retries, `onFailure`, cancellation, and timeouts [[create-function-inngest-documentation]]. RAWR should make `workflow.id` a stable durable key mapped to the Inngest function ID, with `workflow.name` or UI labels treated as display-only.

Triggers should be first-class spec fields. A trigger can be an event or cron, and event triggers can be typed with helper functions and validation [[create-function-inngest-documentation]]. RAWR plugin specs should therefore declare trigger event names, expected payload schema, optional Common Expression Language predicates (CEL, a string expression language over event fields), and whether the event is user-originated, runtime-originated, or cross-plugin. This prevents plugins from burying trigger semantics inside implementation code where the runtime cannot register, validate, document, or test them.

Step identity is just as important as function identity. Inngest uses the first argument of every step method as the step ID, and it uses that ID to memoize step state across function versions. Each step is a discrete task that can be retried, debugged, or recovered independently [[inngest-steps-inngest-documentation]]. RAWR should require every plugin workflow step to have a stable ID and reject IDs derived from array position, timestamps, deploy hashes, install paths, user-facing copy, or other unstable values. A rename should be treated as a durable-state migration requiring explicit versioning policy, because the current vault has stable-ID evidence but not a dedicated migration guide.

The handler body should be deterministic orchestration, not side-effecting plugin code. Inngest executes functions incrementally, persists step results in a function state store, re-enters the handler, skips completed steps, and injects prior step outputs [[how-inngest-functions-are-executed-durable-execution-inngest-documentation]]. Any database call, external API call, filesystem mutation, event send, or time/random-dependent side effect outside a durable step risks being repeated when the handler is re-entered. RAWR should encode the rule directly: top-level handler code may branch on event data and compose steps; side effects belong inside `step.run`, `step.sendEvent`, `step.invoke`, `step.sleep`, or `step.waitForEvent` as appropriate [[inngest-steps-inngest-documentation]].

## IV. Step Boundaries, Retries, And Failure Semantics

`step.run` is the core side-effect boundary. It runs a piece of synchronous or asynchronous code as a retriable step; if it succeeds, the response is saved in function run state and the step will not re-run [[inngest-steps-inngest-documentation]]. In service/spec terms, each RAWR side-effecting step should declare its step ID, side-effect category, idempotency strategy, retry policy, timeout expectations, rollback or fallback path, and observable result shape.

Retries must be explicit because Inngest distinguishes errors from failures. An error causes a step to retry; when attempts are exhausted, the step fails and will not be attempted again in that run; unhandled failures bubble to function failure and cancel future executions [[errors-retries-inngest-documentation]]. The `createFunction` configuration supports `retries` from `0` to `20`, with a captured default of `4`, and supports `onFailure` after all retries [[create-function-inngest-documentation]]. RAWR should decide defaults by workflow class instead of silently inheriting defaults everywhere.

Idempotency must be a reviewable plugin obligation, not just a runtime knob. Inngest states that retrying a step requires the step body to be idempotent, contrasting unsafe inserts with safe upserts [[errors-retries-inngest-documentation]]. RAWR specs should require a written idempotency contract for each side-effecting step: database upsert or unique key, external idempotency key, deduplicated event ID, or compensating cleanup. If a plugin cannot provide one, the step should require either zero retries, an explicit compensating action, or a manual review exception.

Rollback behavior should be step-scoped. A failed step that exhausts retries throws `StepError`, which can be caught to recover, fall back to another service, ignore non-critical work, or run a rollback `step.run`; if the failure is unhandled, the function is marked failed [[rollbacks-inngest-documentation]]. RAWR should encode failure taxonomy as separate layers: HTTP/protocol result from ingress, Inngest run status, Inngest step status, retry attempt count, exhausted `StepError`, handled rollback or fallback result, unhandled function failure, and RAWR payload-level status. Collapsing these into one boolean would recreate the existing spec gap around layer disagreement.

## V. Continuations, Batching, And Flow Control

`step.waitForEvent` should be modeled as a continuation contract. It pauses a run until a named event arrives, returns the received event or `null` on timeout, and supports a matching predicate such as `async.data.userId == event.data.userId` [[wait-for-an-event-inngest-documentation]]. RAWR should require awaited event name, match expression, timeout, null branch behavior, audit expectations, and emitting actor. It should also require race mitigation because Inngest begins listening only when the code reaches the wait; events sent before the wait is executed are not handled by that wait [[wait-for-an-event-inngest-documentation]].

This matters for human-in-the-loop and cross-plugin workflows. A plugin approval flow, external webhook join, or another plugin's completion event is not just a callback. It is a durable continuation with event audit history, a timeout branch, and a possible pre-wait race. RAWR should prefer event-based continuations for fan-out and auditability, but require tests for matched-event continuation, timeout behavior, and "event arrived too early" handling [[wait-for-an-event-inngest-documentation]].

Batching changes both semantics and handler shape. Inngest batching lets one function process multiple events in a single run using `batchEvents` with `maxSize`, `timeout`, optional `key`, and optional `if`; the handler receives `events`, not a single `event` [[batching-events-inngest-documentation]]. RAWR should expose batching only for workflows whose side effects are batch-safe, whose payloads fit memory and size constraints, and whose caller semantics tolerate delayed grouping. The captured docs note a 10 MiB batch safety limit and warn that large batches can cause memory or performance issues [[batching-events-inngest-documentation]].

Batching also needs validation against incompatible controls. The captured batching docs say it cannot be used with idempotency, rate limiting, cancellation events, or priority, and that custom concurrency keys are evaluated against the first event in a batch unless batch and concurrency keys align [[batching-events-inngest-documentation]]. RAWR should enforce those incompatibilities at spec-validation time, not leave them to runtime surprises.

Flow-control primitives must not be treated as synonyms. Concurrency limits executing steps across runs; throttling limits new run starts and queues excess runs for later in first-in, first-out order; rate limiting is a hard limit that skips events beyond the cap; debounce coalesces bursty invocations; priority changes execution order [[flow-control-inngest-documentation]] [[throttling-inngest-documentation]] [[rate-limiting-inngest-documentation]]. The service/spec implication is direct: must-process workflows should generally prefer throttling or concurrency over rate limiting, because rate limiting is intentionally lossy. Rate limiting is appropriate for abuse protection, noisy duplicate webhooks, and low-value synchronization where skipped runs are acceptable [[rate-limiting-inngest-documentation]].

## VI. Testing And Production Caveats

RAWR should split tests into three gates. The first gate is pure spec/conformance: stable function IDs, stable step IDs, no plugin-owned `/api/inngest`, function registry projection, no top-level side-effect patterns, policy field projection, and incompatible batching/control rejection. This can be tested without a live control plane, but it proves only spec shape.

The second gate is vendor-backed local durability. Inngest Dev Server can discover or manually register a local `/api/inngest` endpoint, poll function definitions, call functions directly, accept events through SDK/UI/HTTP, and expose a debug `GET` result showing endpoint configuration [[local-development-inngest-documentation]]. RAWR should add tests that prove a successful `step.run` is memoized, a failed step retries and either succeeds or throws handled `StepError`, `waitForEvent` resumes or times out, and batching/control policy behaves as declared. This is the first gate that can support claims about durable execution behavior.

The third gate is production or production-like operation. Production requires environment keys, app sync, secure ingress, run observation, and deployment-specific readiness. Render guidance, for example, requires setting signing and event keys, syncing the deployed app, and using branch environments for previews [[render-inngest-documentation]]. Connect deployments add app version, instance ID, worker health, graceful shutdown, and worker concurrency checks [[connect-inngest-documentation]]. Self-hosted deployments add a larger control-plane surface: Event API, event stream, Runner, Queue, Executor, state store, database, API, and Dashboard UI [[self-hosting-inngest-documentation]].

Self-hosting is especially important for proof boundaries. The self-hosted architecture shows that durable behavior depends on a queue, runner, executor, and state store that persist initial events, step outputs, and step errors; the default single-binary setup uses SQLite and an in-memory Redis server with snapshots, while external Redis and Postgres are the reliability and scaling path [[self-hosting-inngest-documentation]]. RAWR can use self-hosting for controlled tests, but should not equate a default local single-node proof with production readiness.

Production security caveats should stay explicit. Secrets are environment-scoped, signed ingress must be verified by SDK adapters, sensitive event and step output may need end-to-end encryption middleware, and firewall allowlisting should reference current Inngest IP sources rather than hard-coded ranges [[security-inngest-documentation]]. These are runtime and operator requirements; plugin specs should declare data sensitivity and workflow needs, but they should not own secret storage, route exposure, or network policy.

## VII. Future Implications

The future-proof RAWR design is a manifest-to-runtime compiler for durable workflows. Plugin authors should describe durable intent in a stable, reviewable schema, and the runtime should compile that schema into `createFunction` declarations, step wrappers, policy fields, event schemas, and a controlled function registry. That makes durability inspectable before code runs, which is the only practical way to prevent plugins from smuggling side effects, unstable IDs, lossy rate limits, or unsafe batching into a workflow that operators believe is reliable.

This also gives RAWR a clean evolution path. Today, the runtime can standardize on `serve()` at `/api/inngest` and add Dev Server tests for durable semantics. Later, operators can choose Connect for long-running workers, self-hosted Inngest for controlled infrastructure, or cloud deployment with branch environments, without changing plugin workflow declarations. The plugin contract remains stable because it names durable workflow identity and semantics; the runtime projection changes to fit deployment topology [[serve-inngest-documentation]] [[connect-inngest-documentation]] [[self-hosting-inngest-documentation]].

The strongest architectural recommendation is therefore to encode proof boundaries as part of the spec itself. A workflow can be "declared", "projected", "locally durable-tested", and "production-synced" as separate states. That language prevents RAWR from overstating contained evidence while still turning the current research into concrete implementation requirements. Durable plugin workflows become a service capability with explicit ingress, identity, retries, waits, batching, flow control, keys, and tests, rather than a vague promise attached to background code.
