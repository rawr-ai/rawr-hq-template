The Architecture That Scales
Without Rewriting Itself

Semantic Truth ≠ Runtime Projection ≠ Host Authority

Three distinct boundary types. Four nouns. A semantic kernel that stays stable from a single process to a distributed ecosystem — only placement changes, never meaning.

Three Boundary Types

The architecture is built on the separation of three independent concerns. Scaling changes the deployment boundary without touching the other two.

Semantic What capability IS
Services define this. A service is a contract-bearing capability boundary — transport-neutral, placement-neutral. It doesn't know or care whether it runs in-process or over RPC. This boundary never changes with scale.
Owns: contracts, invariants, business truth Changes at scale: never
Runtime How capability is projected and executed
Plugins project capability into runtime surfaces. Runtime roles (server, async, cli, web, agent) are the kinds of process a host bundle runs — each with its own entrypoint and runtime environment (HTTP for server, durable execution for async, NanoClaw for agent, etc.). They can be colocated in one process or split into individually deployed processes.
Owns: projection, adaptation, surface exposure Changes at scale: colocated → individually deployed
Deployment What gets deployed
The host bundle is a deployable assembly built from one composition authority. It governs which runtime roles exist and what they mount. Roles can share a process or be individually deployed — but they all answer to the same composition authority. Promotion creates a new host bundle with its own composition authority.
Owns: boot, lifecycle, config, composition Changes at scale: bundle count (via promotion)

How It Scales

Start simple without lying about what the system is. Each scale step changes a specific boundary type while preserving the others.

n = 1

Colocated — One Bundle, Shared Process

One host bundle, one composition authority. Runtime roles share a process. Services called in-process. This is real architecture, not a temporary monolith.

Split Roles — Same Composition Authority, Separate Processes

Roles split into individually deployed processes for operational reasons. Still one host bundle, one composition authority governing all of them. The runtime boundary changed; the semantic and composition boundaries did not.

PROMOTE

Promote — New Deploy Unit, Peer Host Bundle

When a domain outgrows HQ, it promotes into its own host bundle with its own composition authority. This is the scale step that creates a new composition authority — a new peer of HQ, not a child. The promoted bundle carries the same four-kind structure.

● Semantic boundaries: unchanged
▲ Deploy units: 1 → 2
▲ Composition authorities: 1 → 2

See It at Full Scale

Multiple domains promoted to peer bundles. The semantic kernel sits at center. Each host bundle is a deploy unit with the same internal structure. Outer bundles can promote their own sub-domains — the pattern is recursive.

Gold center = semantic kernel (services + packages). Each coral node = one deploy unit (host bundle) with its own packages/ services/ plugins/ apps/.

Why This Shape Exists

Strong nouns with distinct boundary types ↓ Scale splits runtime or composition boundaries — never semantic ones ↓ Coherent systems at any scale ↓ Leverage, not tidiness

The architecture reduces ambient ambiguity so that later systems — enforcement, scaffolding, observability, and eventually autonomous stewardship — can be built on top of it without fighting the structure at every layer.
