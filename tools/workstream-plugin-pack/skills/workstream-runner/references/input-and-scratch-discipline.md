# Input And Scratch Discipline

Treat inputs as typed control surfaces, not undifferentiated context.

| Input kind | Meaning | Effect |
| --- | --- | --- |
| Opening input | Creates or materially reframes the workstream | Defines objective, target, and output contract |
| Authority input | Can decide what is true or required | Wins or loses according to authority order |
| Coordination input | Helps resume or coordinate work | Informs execution but does not decide truth by itself |
| Evidence input | Supports or weakens claims | Can justify findings, repairs, or replanning |
| Stale/excluded input | Must not be promoted without revalidation | Stays fenced |
| Control input | Approves, pauses, retargets, abandons, or closes | Changes posture only when explicit |

Most context is evidence or coordination, not control. Do not treat old
transcripts, scratch notes, generated summaries, or historical phase artifacts
as authority unless the DRA explicitly accepts them.

Scratch documents are running notes. They are not proof, not final workstream
artifacts, and not authority. Before closure, every scratch surface is deleted,
archived as provenance, quarantined with a non-authority header, or integrated
into a real artifact.

Process feedback is allowed at closure when the workflow itself caused
friction. Record it as a concrete workflow/process repair or deferred item, not
as vague commentary.
