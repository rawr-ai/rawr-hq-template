## Width Findings For Parent Capture

1. The source set should support a clear spec boundary: plugin authors declare workflow descriptors, triggers, IDs, step IDs, flow controls, batching, waits, and retry/failure policy; the runtime owns the Inngest client, function registration, ingress route, signing-key environment, and Dev Server/production verification posture.
2. The RAWR lab proof should remain classified as contained boundary evidence. Official docs show that durable semantics require Dev Server or platform behavior, signed ingress, function registration/sync, event delivery, retry/error semantics, and run-state observation that the lab did not yet prove.
3. `/api/inngest` is the highest-risk conflation point: it is an Inngest runtime ingress endpoint for GET diagnostics/metadata, POST invocation, and PUT sync/registration, not a public workflow trigger API. RAWR trigger APIs should send events or call controlled runtime operations instead.
4. `step.run` is the main plugin workflow unit boundary. RAWR specs should require named, stable, idempotent steps for non-deterministic work and define how `StepError`, rollbacks, retries, and layer-disagreement status are observed.
5. Flow controls and batching are not incidental implementation options. They affect correctness: throttling delays, rate limiting skips, batching changes handler input to `events`, and batching has feature incompatibilities and memory/size caveats.
