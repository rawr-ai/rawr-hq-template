# Source Capture Log - Official Inngest Docs Width Sweep

Research query: For RAWR HQ runtime realization, which Inngest primitives and operational semantics should the plugin/runtime specs encode for durable plugin workflows?

Scope note: the local RAWR files are grounding context for what the runtime-realization lab has and has not proven. They are not vendor authority. Vendor authority in this packet is limited to official `https://www.inngest.com/docs/...` documentation.

## Official Source Matrix

| Official Inngest URL | Vendor claim focus | RAWR relevance | Caveats / capture notes |
| --- | --- | --- | --- |
| https://www.inngest.com/docs/reference/serve | `serve()` exposes application functions over HTTP, receives metadata/invocation/sync requests, accepts `{ client, functions }`, recommends `/api/inngest`, and expects signing key configuration on the client/environment. | Confirms the RAWR async host boundary should be a runtime-owned Inngest bundle mounted at `/api/inngest`, not a plugin/browser API surface. | RAWR lab proved only local `inngest/bun` handler passage; this source supplies normative endpoint semantics and security-sensitive method roles. |
| https://www.inngest.com/docs/reference/typescript/functions/create | `inngest.createFunction()` defines durable functions with stable `id`, triggers, retries, `onFailure`, flow-control options, `batchEvents`, `cancelOn`, and timeouts. | Plugin workflow specs should encode stable function IDs, trigger event names, retry policy, idempotency/flow-control choices, and failure handling as declared workflow contract metadata. | Local RAWR evidence currently exercises function shape, not durable scheduling or policy enforcement. |
| https://www.inngest.com/docs/learn/inngest-steps | Steps are durable workflow units; step IDs memoize state; non-deterministic side effects belong in `step.run`; successful `step.run` results are saved and not re-run. | RAWR plugin boundaries should treat each runtime-owned side effect as a stable `step.run` boundary, with step IDs as public/semipublic compatibility keys. | Local proof crossed `step.run`, but did not prove persisted memoization across durable retries. |
| https://www.inngest.com/docs/learn/how-functions-are-executed | Inngest functions execute incrementally; step results/errors are persisted outside the function execution context; successful steps are memoized and skipped on resume. | Specs must distinguish handler re-entry from step execution, and tests must avoid top-level side effects outside durable step boundaries. | This source is central vendor authority for durable semantics the RAWR lab explicitly did not prove. |
| https://www.inngest.com/docs/guides/error-handling | Inngest retries errors automatically; exhausted retries fail the step/run; retried step code must be idempotent. | RAWR specs should require idempotent step bodies, explicit retry counts, layer-specific failure classification, and sanitized failure observation instead of treating HTTP success as runtime success. | Complements local layer-disagreement findings: transport/protocol success and RAWR runtime failure can diverge. |
| https://www.inngest.com/docs/features/inngest-functions/error-retries/rollbacks | Rollback/failure handling is part of Inngest's error/retry model. | Workflow specs should separate normal durable steps from compensating actions/failure hooks and decide which failures are compensatable. | Used as error-handling coverage; do not claim RAWR has tested rollback behavior. |
| https://www.inngest.com/docs/features/inngest-functions/steps-workflows/wait-for-event | `step.waitForEvent()` pauses a run until a matching event arrives or timeout returns null; it begins listening when executed and does not catch earlier events. | Human-in-the-loop and cross-plugin workflows should model wait-event names, match predicates, timeout behavior, timeout branches, and race checks explicitly. | Race caveat is important for RAWR specs: do not encode waits without event-ordering proof or compensating state checks. |
| https://www.inngest.com/docs/guides/flow-control | Function-level flow-control primitives include concurrency, throttling, rate limiting, debounce, and priority with distinct operational semantics. | RAWR specs should make flow-control a workflow boundary policy, not an incidental implementation option. | The archived spec-gap assessment already identified this as missing; this doc confirms it is implementation-critical. |
| https://www.inngest.com/docs/guides/throttling | Throttling limits how many function runs start in a time period and queues excess runs FIFO for later capacity. | Use throttling when RAWR must process all workflow events but smooth starts or respect vendor limits. | Throttling controls function starts, not step concurrency. |
| https://www.inngest.com/docs/guides/rate-limiting | Rate limiting is a hard limit; excess matching events are skipped/ignored for function execution, though events remain stored. | Use rate limiting only when lossy behavior is acceptable, e.g. abuse/noisy duplicate protection, not for must-process plugin workflow events. | This distinction should be explicit in RAWR workflow policy because misuse silently drops work. |
| https://www.inngest.com/docs/guides/batching | `batchEvents` lets one function run process multiple events; batches have `maxSize`, `timeout`, optional key/if; batch size is limited and batching conflicts with idempotency, rate limiting, cancellation events, and priority. | RAWR plugin specs should expose batching only for batch-safe operations and require batch key/memory-limit reasoning plus incompatible-policy checks. | Local RAWR evidence does not test batch mode; use as vendor guidance only. |
| https://www.inngest.com/docs/local-development | Dev Server discovers or is given `serve` URLs, polls local apps, calls functions directly, supports UI/test events, and local keys can be dummy/non-validated. | RAWR local tests should distinguish hermetic unit tests, local Dev Server integration tests, and production-key tests. | Local dev behavior is not production security proof; do not infer key/signature guarantees from dev mode. |
| https://www.inngest.com/docs/learn/security | Requests between Inngest and app servers are signed with environment-specific signing keys; SDK `serve` adapters verify signatures and timestamps; keys must remain secret and can be rotated; event/API keys have separate roles. | RAWR `/api/inngest` must be treated as signed runtime ingress with secret management, rotation, gateway/firewall allowlisting, and no browser/API-origin access. | Local RAWR lab evidence is not a substitute for signed ingress verification or production gateway policy. |
| https://www.inngest.com/docs/setup/connect | `connect` creates outbound persistent worker connections, supports horizontal scaling, graceful shutdown, and production requires signing/event keys, app versions, instance IDs, and concurrency settings. | RAWR specs can keep `serve()` as the initial ingress model but should reserve a production topology decision for `serve` vs Connect workers. | Connect is public beta in the captured doc; do not make it mandatory without a separate production topology decision. |
| https://www.inngest.com/docs/deploy/render | Production deployment requires serving Inngest functions, setting `INNGEST_SIGNING_KEY` and `INNGEST_EVENT_KEY`, syncing the app, and separately handling preview/branch environments. | RAWR deployment caveats should require sync/version/environment handling for each runtime environment, not only code mount correctness. | Render is an example cloud deployment, not RAWR's deployment authority. |
| https://www.inngest.com/docs/self-hosting | Self-hosted Inngest includes Event API, Runner, Queue, Executor, state store, DB, API, and UI; starts with event/signing keys; default local persistence is SQLite/in-memory Redis unless external services are configured. | Useful for understanding durable-state components and for local/lab topology caveats; RAWR should not imply durable behavior without a running Inngest control plane/state store. | Self-hosting has support caveats and scaling/storage choices; local single-node defaults are not production HA proof. |

## Width Verdict

The official docs support a stronger spec posture than the current RAWR lab evidence alone: RAWR already has meaningful contained boundary passage for `serve`/`createFunction`/`step.run`, but durable plugin workflow specs must additionally encode stable function and step identity, idempotent retriable side effects, wait-event race semantics, batching and flow-control policy, local-vs-production environment distinctions, signed `/api/inngest` ingress, app sync/versioning, and production topology caveats.
