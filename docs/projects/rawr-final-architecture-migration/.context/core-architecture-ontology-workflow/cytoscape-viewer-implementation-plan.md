# Cytoscape Graph Viewer Operationalization Plan

## Summary

Replace the current list-style ontology HTML with a real Cytoscape.js graph explorer generated by `semantica:core:visualize`. The authoritative inputs remain the reviewed YAML ontology and generated JSON graph artifacts; Cytoscape is only the local/online inspection UI.

The implementation should produce a self-contained HTML viewer that opens directly from `.semantica/current/graph-viewer.html`, plus an optional local serve command for a cleaner browser workflow. Cytoscape is the visualization surface because it accepts JSON-style graph elements and supports layouts, selectors, events, filtering, and graph traversal in-browser.

The implementation also includes an explicit semantic-query surface so agents can ask RDF/SPARQL-style questions against the generated graph without needing Neo4j, Fuseki, or another persistent graph service in this pass.

## Phase 0: Branch, Plan Capture, And Grounding

- Work on a new stacked Graphite branch above `codex/core-ontology-operationalization`: `codex/cytoscape-ontology-viewer`.
- First implementation action: write this plan to this file.
- Add a short orchestrator/team workflow at `cytoscape-viewer-orchestrator-workflow.md`.
- Preserve the existing ontology/runtime/diff outputs under ignored `.semantica/`; do not treat generated graph state as source truth.

## Phase 1: Graph Data Contract For Visualization

- Add a deterministic transformation from `layered-graph.json` / `document-diff.json` into Cytoscape elements.
- Nodes carry `id`, `label`, `type`, `layer`, `status`, `definition`, `source_refs`, `operational_consequence`, and `classifier_readiness`.
- Edges carry `id`, `source`, `target`, `predicate`, `layer`, `status`, `source_refs`, and `operational_consequence`.
- Diff overlays are keyed by canonical entity id where possible.
- Derived views: `canonical`, `layered`, `runtime`, `authority`, `testing-plan-diff`.
- Candidates and evidence-only terms remain visually distinct and hidden by default.

## Phase 2: Cytoscape Viewer Implementation

- Add Cytoscape as a repo dependency managed by Bun.
- Generate a self-contained HTML file by embedding the Cytoscape browser bundle and graph payload directly into `graph-viewer.html`.
- Replace the table-only viewer with a full-screen graph canvas, control panel, details panel, and status area.
- Support `cose`, `breadthfirst`, `concentric`, and `grid` layouts.
- Add click inspection, double-click focus, neighborhood mode, path mode, reset, fit, export PNG, and copy selected item JSON.
- Style nodes by layer/type and edges by predicate/status; make forbidden/deprecated/review-needed states visually obvious without implying they are canonical truth.

## Phase 3: Throughline Views And Diff Review

- Add curated throughline presets: ownership and authority, runtime realization lifecycle, gates and validation, forbidden/replacement terms, testing-plan alignment, classifier-readiness candidates.
- Each throughline applies deterministic filters plus layout choice instead of relying only on force-directed layout.
- Integrate testing-plan diff findings into the viewer: aligned, review-needed, underrepresented gates, and future stale/forbidden findings.
- Display repo-relative source references, line spans, and sections.

## Phase 4: Local UI And Agent Query Surface

- Keep `bun run semantica:core:visualize` as the main command and make it print the generated HTML path.
- Add `semantica:core:serve` / `core:serve` to serve the selected run directory read-only, defaulting to `.semantica/current`.
- Add `semantica:core:query` / `core:query` for agent-oriented graph questions.
- Support both named JSON graph queries and SPARQL files against `semantica-data-graph.ttl` when RDFLib is available.
- Ship tracked query examples so agents can see the intended question shape.
- Do not introduce Neo4j, RDF stores, or SPARQL servers in this implementation; keep existing GraphML/RDF/OWL/SHACL exports as adjacent inspection/query outputs.

## Phase 5: Team Strategy And Review Gates

- Use a small agent team only where it reduces risk: data-contract review, viewer behavior review, semantic-query review, verification review.
- Codex remains DRI for final synthesis, branch hygiene, and deciding what lands.
- Every agent gets the same framing: YAML ontology is authoritative, generated graph is derived, candidates/evidence must not be promoted visually, and the goal is decision-grade inspection.
